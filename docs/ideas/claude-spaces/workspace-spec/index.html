Agent Workspace Specification

The Agent Workspace Model defines a standard pattern for turning any Git repository into a persistent, context-aware AI workspace. This page documents the complete specification — the folder structure, conventions, and components that make it work.

Reference Repository: The full specification, validation schema, and examples live in the Claude Agent Workspace Model repo.

Point Claude at that repo and tell it to scaffold a workspace for your use case — it will generate the full structure automatically.

Lifecycle Stages

Every workspace moves through three stages:

1. Scaffold Clone template, get the structure
2. Personalize Run /onboard, ingest context
3. Ready Fully configured, start working

The cloned template comes pre-populated with the structural mechanics (folders, CLAUDE.md, slash commands, MCP config) but is a blank page in terms of context. The /onboard command drives the Personalize stage.

Required Directory Structure

Every Agent Workspace must contain these directories and a root CLAUDE.md:

my-workspace/ ├── CLAUDE.md # Agent instructions (keep lightweight) ├── context/ # User-provided context and agent reference │ ├── project.md # Project name, description, goals │ ├── role.md # User's role and how it shapes the work │ ├── constraints.md # Boundaries, deadlines, preferences │ └── for-agent/ # Detailed instructions for the agent │ ├── environment.md # OS, tools, paths │ ├── workflows.md # Step-by-step workflow instructions │ └── ... ├── work-log/ # Daily operation logs (YYYY-MM-DD.md) ├── planning/ # Active plan + pivot records │ ├── plan.md # Current active plan │ └── pivots/ # Plan change records ├── user-docs/ # Agent-produced deliverables for the user ├── .claude/ │ └── commands/ # Slash commands │ └── onboard.md # Required: initializes the workspace └── .mcp.json # Optional: MCP server configurations

Component Reference

CLAUDE.md — The Agent's Instruction File

This file is loaded with every prompt, so it must be kept lightweight. It should contain only:

  • What this workspace is and what template it was created from
  • The agent's role (one sentence)
  • A list of available slash commands
  • Stubs pointing to detailed context files

Detailed instructions, environment specifics, and domain knowledge belong in context/for-agent/, not in CLAUDE.md.

# Competitive Research Workspace You are a competitive research analyst. This workspace was created from the competitive-research template. ## Context References - For user environment details, see: context/for-agent/environment.md - For research methodology, see: context/for-agent/workflows.md - For project scope, see: context/project.md ## Commands - /onboard — Initialize this workspace with project context - /report — Generate a competitive analysis report

context/ — The Knowledge Base

PathPurposeCreated By
project.mdProject name, description, goals/onboard
role.mdUser's role and how it shapes the work/onboard
constraints.mdBoundaries, deadlines, preferences/onboard
for-agent/environment.mdUser's environment details (OS, tools, paths)/onboard
for-agent/workflows.mdDetailed workflow instructions for the agentTemplate or /onboard
for-agent/*.mdAny detailed agent instructions too verbose for CLAUDE.mdTemplate, user, or agent

Key principle: CLAUDE.md contains stubs; context/for-agent/ contains substance. The agent reads detailed files on demand rather than loading everything into every prompt.

work-log/ — Operational History

The agent creates a new entry for each day it operates. Entries document what was done, what was decided, and what is pending. Each file is named YYYY-MM-DD.md.

planning/ — Plans and Pivots

There is always exactly one plan.md representing the current state. When the plan changes significantly, the agent records the pivot in pivots/ with the date and reason before updating plan.md.

user-docs/ — Deliverables

Polished, user-facing documents: research summaries, decision frameworks, reference guides, or anything the user asked the agent to produce.

.claude/commands/ — Slash Commands

Slash commands define repeatable workflows. The /onboard command is required — it gathers context and initializes the workspace. Additional commands are domain-specific.

# /command-name Description of what this command does. ## Steps 1. First, do this 2. Then, check that 3. Finally, report results ## Parameters - $ARGUMENTS: Optional input from user

The Four-Layer Stack

Agent Workspaces are built from four layers that work together:

1 Context Layer

CLAUDE.md + context/ directory + working files. The foundation — what makes it a "workspace" rather than just a CLI session. Project CLAUDE.md takes priority over global CLAUDE.md.

2 Command Layer

Slash commands (.claude/commands/), shell scripts (scripts/), and context aliases. These encode repeatable workflows and shortcuts specific to the workspace domain.

3 Integration Layer

MCP servers, external APIs, CLI tools. Configured via .mcp.json to connect the workspace to databases, cloud services, local tools, and communication channels.

4 Execution Layer

File operations, shell commands, tool calls. How the agent acts on the world — reading, writing, editing files and invoking external services.

Memory Management

The repository is the memory store. CLAUDE.md should instruct the agent to prioritize the repository's own context files over any built-in memory features. This ensures persistence and portability — the repository travels with the project.

# Example instruction for CLAUDE.md Use this repository as your primary memory. Do not rely on built-in memory features for project context. All persistent context belongs in context/. All operational history belongs in work-log/.

Validation Checklist

An agent entering a workspace should verify:

  • CLAUDE.md exists at root
  • context/ directory exists
  • work-log/ directory exists
  • planning/ directory exists
  • user-docs/ directory exists
  • .claude/commands/onboard.md exists
  • If context/project.md does not exist, the workspace has not been onboarded — prompt the user to run /onboard
